## 需求分析
---

最新消息幾乎是每個網站必備的功能，一般來說，需求是這樣的：

後台上稿，前台顯示。
可以附加檔案。
在後台上稿的部份，有二種做法，一種是只輸入純文字，另一種是使用HTML編輯器，這兩種做法各有優缺點。

**上稿：使用HTML編輯器**

使用HTML編輯器才能夠編輯較豐富的內容，例如加入連結、改變字體大小、控制字體顏色，甚至加入圖片，這些功能在多數人平常都用慣了Word之後，變成是編輯器必要的功能，因此這也會是多數業主的選擇，但自由度越大，功能越多，衍生的問題也就會越多，常見的問題有：

無障礙網站，但因後台上稿內容而無法通過無障礙檢測。
後台上稿內容破壞了前台版面。
因為目前無法在上稿的同時，由程式即時進行無障礙檢測，所以，如果網站需要長期符合無障礙標準，只能請負責上稿的人員多加留心注意上稿內容是不是符合無障礙標準，不過這項對於上稿人員的要求，事實上有執行上的困難，因為上稿人員通常不會知道無障礙標準。

至於上稿內容破壞前台版面的部份，最常發生在插入的圖片尺寸過大時。在html編輯器裡插入圖片有二種做法，一種是插入圖片的連結，一種是直接上傳圖片，如果使用的是插入圖片連結，必須由上稿人員自行設定圖片大小以符合前台的版面，如果是直接上傳圖片，則可由程式進行縮圖。

另一種比較不常發生，但是也存在的可能性是上稿內容的html原始碼格式錯誤，在前台顯示時，造成整個前台頁面的原始碼錯誤，因而版面出現問題，這部份的問題，只能採用亡羊補牢的做法，也就是看見版面出問題之後，再經由檢視原始碼去找出並解決問題。

**上稿：使用純文字**

使用純文字，相對於使用HTML編輯器，問題要少得多，很單純的後台輸入什麼，前台就顯示什麼就行了。主要的問題在無法完全滿足需求，如圖片、連結，這部份可利用在上稿編輯的頁面中，多提供圖片上傳及輸入相關連結的欄位來克服。

附加檔案

附加檔案這功能的考量重點是檔案的數目是固定的還是可變動無上限的。這二個不同的要求，會大大的影響程式的實作。


## 資料庫設計
---

在考量過所有的需求後，可以歸納出最新消息這功能，在資料庫的部份，會需要二個table來記錄相關資料，其中一個記錄的是最新消息本身的資料，另一個table記錄的是附加檔案的資料。

每一則最新消息，至少會包含有主旨、內容、新增日期，選擇性的資訊則有上架時間、下載時間；在附檔的資料部份，包含的有：對應的最新消息編號、檔案名稱、檔案大小、下載網址。

最後可以建立起如下的二個table


以MySQL來建立資料庫的話 ，SQL的語法為：

最新消息

{% highlight sql %}
            
  CREATE TABLE `news` (
      `id` bigint(20) unsigned NOT NULL AUTO_INCREMENT,
      `source` varchar(255) DEFAULT NULL,
      `title` varchar(255) NOT NULL,
      `content` text, 
      `start_date` datetime DEFAULT NULL,
      `end_date` datetime DEFAULT NULL,
      `created_on` datetime NOT NULL,
      `updated_on` datetime DEFAULT NULL,
       PRIMARY KEY (`id`)
  ) ENGINE=MyISAM

{% endhighlight %}



最新消息附加檔案


{% highlight sql %}
  CREATE TABLE `news_attach` (
      `id` bigint(20) unsigned NOT NULL AUTO_INCREMENT,
      `news_id` bigint(20) NOT NULL,
      `filename` varchar(128) DEFAULT NULL,
      `size` bigint(20) DEFAULT '0',
      `uri` varchar(255) DEFAULT NULL,
      `created_on` datetime DEFAULT NULL,
       PRIMARY KEY (`id`)
  ) ENGINE=MyISAM

{% endhighlight %}

到此，資料庫的設計就算完成了

有了DAO後，另一個重點是在MVC的controller端，

在後台的Controller應該俱備有列表、新增、編輯、儲存、刪除，這些功能。
